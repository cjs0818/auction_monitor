# 법원경매 검색 고속화 수정

## 확인된 병목

법원 목록조회는 90일 기준 14일 단위로 약 7회 요청하지만, 기존 버전은 목록검색 이후에도
보고서 상위 20건 각각에 대해 다음 작업을 순차 수행했습니다.

1. 사건상세 JSON 재조회로 다음 매각기일·최저매각가격 교정
2. 사건검색 화면 재진입
3. 물건행 및 현황사진 탭 탐색
4. 사진이 없으면 네이버 위치지도 로딩·캡처
5. 정상 응답과 사진 실패 화면의 디버그 파일 저장

따라서 목록검색보다 상위 후보 상세보강 시간이 더 길어질 수 있었습니다.

## 적용한 개선

### 1. 목록값 기반 스마트 가격교정

법원 목록응답에 `notifyMinmaePrice1~4`의 다음 회차 가격과 매각기일이 모두 있으면
사건상세 가격 재조회를 생략합니다. 가격이나 기일이 없거나, 유찰물건인데 구형
`minmaePrice`만 남은 경우처럼 교정 필요성이 큰 물건만 상세조회합니다.

- 기본 상세교정 신규조회 상한: 6건
- 고속모드를 끄면 기존처럼 상위 후보 전체를 교정할 수 있음
- 카드와 DB에는 목록가격 사용 여부와 상세교정 여부를 계속 기록

### 2. 사진 신규수집의 점진적 캐시

- 기존 캐시 사진·위치지도: 수량 제한 없이 즉시 재사용
- 캐시에 없는 신규 사진: 기본 실행당 최대 6건
- 다음 검색에서는 앞서 저장된 6건을 즉시 사용하고 다음 미수집 후보를 추가 수집
- 가장 빠른 검색이 필요하면 설정에서 신규 사진 수집 상한을 0으로 지정 가능

### 3. 사진 탭 탐색 단축

현황사진·물건사진 등 6개 라벨을 하나씩 전체 DOM에서 찾던 방식을 한 번의 우선순위
탐색으로 변경했습니다. 사진 캡처 대기시간도 최대 2.5초로 제한했습니다.

### 4. 사건검색 화면 재사용

직전 물건 처리 후 사건검색 폼이 남아 있으면 동일 페이지를 재사용합니다. 매 물건마다
사건검색 URL을 다시 여는 불필요한 전체 페이지 로드를 줄였습니다.

### 5. 실패 음성 캐시

현황사진도 없고 위치지도 생성도 실패한 물건은 기본 7일간 실패 상태를 저장해 같은
물건을 검색할 때마다 느린 화면 탐색을 반복하지 않습니다.

### 6. 정상 실행 디버그 I/O 제거

기존에는 정상 법원 요청에서도 큰 JSON 응답을 여러 파일로 중복 저장하고, 사진이 없는
물건마다 화면 PNG와 HTML을 저장했습니다. 기본값에서는 이를 끄고 오류 발생 시에만
진단자료를 남기도록 변경했습니다.

## 기본 설정

```yaml
fast_mode: true
price_detail_policy: smart
price_detail_max_per_run: 6
detail_min_delay_seconds: 1.5
detail_jitter_seconds: 0.5
photo_max_per_run: 6
photo_wait_seconds: 0.45
photo_capture_timeout_seconds: 2.5
photo_missing_cache_days: 7
save_exchange_debug: false
photo_debug_enabled: false
```

목록검색 요청은 법원 사이트 보호를 위해 기존의 3.0초 + 최대 1.5초 무작위 대기를
유지합니다. 고속화는 주로 중복 상세조회, 사진 화면 탐색 및 디스크 저장을 줄이는 방식으로
적용했습니다.
